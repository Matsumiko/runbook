# Summary And Action Hierarchy

Build the top of a detail page to answer the main questions quickly:

1. what this thing is
2. what state it is in
3. what the user can do next

Prefer a summary structure like:

1. breadcrumb or route context if the repo already uses it
2. record title or identity block
3. status, badges, or metadata with obvious priority
4. primary action and carefully subordinated secondary actions
5. supporting stats or identity details only if they help the next decision

Keep rules like these in mind:

- do not let every metadata point have the same visual weight
- keep destructive actions separated from routine actions
- keep the primary action visible near the summary, not buried lower in the page
- use badges, icons, and metadata rows to clarify, not decorate
- on mobile, preserve title, status, and primary action before dense supporting data
