# No Redesign Rule

This skill is for refinement, not reinvention.

## Keep These Boundaries

- Do not swap the visual language of the product.
- Do not change the component family just because you prefer another style.
- Do not turn a polish pass into a broad refactor.
- Do not introduce large motion, gradient, or layout changes unless the user explicitly asks for them.

## What Counts as Legitimate Polish

- cleaner hierarchy
- better spacing and alignment
- stronger states
- more coherent responsive behavior
- removal of obvious visual inconsistencies
- improved readability and scanability

## What Usually Counts as Redesign

- replacing the component style direction
- changing the entire page composition
- introducing a new token system
- swapping the tone of the product from dense to airy or vice versa without instruction
- restyling every component family to match a different aesthetic

## Escalation Rule

If the real issue is architectural or systemic rather than polish:

- state that clearly
- do the smallest safe refinement you can
- report the deeper issue as a follow-up instead of hiding it
