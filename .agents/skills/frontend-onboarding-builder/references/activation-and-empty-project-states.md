# Activation And Empty Project States

Treat onboarding completion as the handoff into real product use.

Common activation moments include:

- first workspace created
- first project or record created
- first teammate invited
- first integration connected
- initial import started or completed
- primary preferences chosen

Use rules like these:

- after each step, make the next high-value action obvious
- if setup is complete but the product is still empty, show a guided empty state instead of a dead end
- if external setup blocks progress, explain the block and the recovery path clearly
- completion should feel like entry into the product, not just a congratulation screen
- if users can resume later, preserve enough context that they do not feel reset

Non-happy-path checklist:

- loading state does not hide the current step or goal
- skipped setup still leaves a viable path back into configuration
- blocked integrations or imports explain what to do next
- empty post-onboarding states still point toward first value
- narrow layouts keep progress, current step, and primary action readable
