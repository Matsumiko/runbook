# Acceptance Checklist

Use this as the last pass before calling the component work done.

## System Fit

- Does it match the existing stack and project conventions?
- Does it reuse existing tokens or theme values where possible?
- Does it feel related to nearby components?

## API Quality

- Is the component contract understandable?
- Are variant names and prop names consistent with the repo?
- Is the abstraction level appropriate?

## UX Quality

- Are hierarchy and spacing clear?
- Are key states implemented?
- Is responsive behavior acceptable where needed?
- Does the component avoid visual noise?

## Accessibility

- Is keyboard interaction handled where relevant?
- Is focus visible?
- Are semantics and disabled states preserved?
- Are labels and status messages understandable?

## Verification

- Were available checks actually run?
- Was the component seen in at least one realistic context?
- Were unverified areas stated honestly?

## Done Means

The component is only done when it is system-aligned, state-aware, and honestly verified.
